## 0. 现状与病灶

现状截图：`assets-2026-07-19/10-current-1920.png`（1920×1080，6 个项目、215 字文案）

```
项目列表 180 │ 设置 200 │ 预览 300-380 │ 文案 + 字幕 1fr      ← 整体限宽 1200 居中
```

截图里能直接读出四个问题，其中三个正是用户抱怨的：

1. **项目列表 180px × 整屏高，装的是 6 行东西。** 竖直方向 90% 是空的。
   它占掉了 15% 的工作台宽度，来换一个「一天点两次」的操作。
2. **设置栏 200px 太窄。** BGM 选项里「一笑倾城 现言 甜文 7:33」已经在挤，
   曲名、标签、时长三段在 200px 里互相顶。滑块也短得不好拖。
3. **字幕列表被压在文案栏下半。** 文案只用了上半栏的 40%，
   而字幕列表（18 行）在下半栏里要滚动——两边都不舒服，只因为它们被迫共享一栏。
4. **预览栏 300-380px，但预览框限宽 `26vh`（1080 屏上约 280px）。**
   预览框左右各余一条空隙，下方到导出按钮之间是一大片死区。

## 1. 项目列表怎么办

### 推荐：**收进文案列头，做成项目切换器（popover）**

关键观察：**文案栏的列头现在显示的就是项目名**（`Workspace.tsx:274`，`{project?.name ?? '选一个项目开始'}`）。
也就是说「我在哪个项目」这件事**已经有位置了**，而且位置正确——它是这一屏的标题。
那就不必再花一整栏重复说一遍，只要把那行字变成可点的按钮。

**零新增 chrome**：不加顶栏、不加图标条，列头高度仍是 h-14，和其余三列的头部对齐成同一条带。

截图：`assets-2026-07-19/50-popover.png`

```
┌ 文案列头 ────────────────────────────
│  SUREJACK   自律的真相  ⌄
└──────────┬──────────────────────────
           │  + 新建项目
           ├──────────────────────
           │ ▎自律的真相        215 字   ← 当前项，琥珀左条 + ink-700
           │   为什么你总是很累   70 字
           │   存钱的三个误区    100 字
           │   早起这件事       130 字
           │   如何读完一本书    160 字   ← hover 时右侧浮出删除图标
           │   朋友圈的社交成本    88 字
           └──────────────────────  300px 宽，列表段 max-h 340 可滚
```

取舍：

- **「我在哪个项目」永远可见** —— 不是"可以查到"，是它就写在标题位上，比原来那栏的
  高亮行还醒目（原来那行只是列表里一个 ink-700 底的条目）。
- **切换成本从 1 次点击变成 2 次** —— 用户自己说了切项目频率远低于编辑。
  换 180px × 100% 的常驻面积，划算。
- 新建 / 删除都收进 popover，`ProjectList.tsx` 的逻辑基本原样搬进去，改的是容器不是行为。
- `SUREJACK` 字标降级成 12px / `ink-600` / `letter-spacing .14em` 的小字，
  只做产品身份锚点，不参与视觉层级。
- **用户/登出/版本角标**搬到最右侧设置列的底部（原来在项目栏底部）。
  位置对称，且它和「设置」同属低频的账户/系统区。

### 被否决的备选

**备选 A：收成 56px 图标窄条。** 否决——**项目没有图标**。
唯一能塞进 56px 的是项目名首字（「自」「为」「存」「早」）。
6 个单字方块既认不出是哪个项目，又在深色 UI 里排成一列很脏。
现有代码的折叠态（`collapsed`）之所以能接受，正是因为它折叠后**什么都不显示**，
只留一个展开箭头——那说明这栏折起来时是纯粹的浪费，不如彻底删掉。

**备选 B：和设置栏合并成可切页侧栏（「项目 / 设置」两个 tab）。** 否决——
把**高频**的东西藏到**低频**的东西后面。设置里的字幕高度、BGM、音量是编辑过程中反复调的，
而项目是一天切两次的。做成 tab 意味着每次调音量前都可能要先切回「设置」页。方向反了。

**备选 C：顶栏横排项目 chip 条。** 部分可取但否决——一眼能看全所有项目是真优点，
但 6–10 个中文项目名（「朋友圈的社交成本」8 个字）横排必然溢出要横向滚动，
而且它在整屏最顶上加了一条常驻的、花花绿绿的视觉噪音。
popover 里的竖排列表同样一眼看全，只是多一次点击。

### 推荐排布

```

```

截图：`assets-2026-07-19/60-zone-950.png`（推荐版）、`40-final-1920.png`（字幕列用 ink-900 的对照版）

**这确实是一次「设置 ↔ 字幕」互换**：设置原本在预览左侧那个槽位，现在那个槽位归字幕；
字幕原本在最右那一列，现在最右那一列归设置。两者都是 280px，等宽。

**为什么是这个顺序**：从左到右正好是制作流水线 —— **写（文案）→ 听（配音·字幕）→ 看（预览）→ 调（设置）**。
而且设置和预览相邻这件事是有功能理由的：字幕高度、BGM、音量**改的就是预览里看到/听到的东西**，
因果必须在同一个视野里（这条约束在现状里也成立，方案没有把它弄丢）。

### 互换之后，文案和字幕的关系怎么保持

用户的顾虑是对的：字幕是配音派生的，和文案是一回事，不能因为拆栏就拆散。三条手段：

1. **它们仍然紧邻**，只是关系从「上下」变成「左右」。
   文案在 ①、字幕在 ②，中间只隔一条 `border-line`。
2. **共用最深的表面 `ink-950`。**（关键手法，见 `60-zone-950.png` 与 `40-final-1920.png` 对比）
   ①② 用 `ink-950`，③④ 用 `ink-900`。左半屏因此读作**一整块连续的内容区**，
   右半屏读作工具区——四栏在结构上是四列，在视觉上是 **2 + 2**。
   这比在它们外面画一个框便宜得多，也克制得多，且不引入任何新颜色。
3. **配音控件留在字幕列头部**（现在就在 `SubtitleList.tsx` 里调 `VoicePanel`，不用动）。
   这样第 ② 列自述为完整的一句话：「配音在这儿生成 → 字幕从它掉下来」。
   文案 → 配音 → 字幕的因果链，从左到右读得出来。

### 为什么等宽是 280 而不是 200 或 320

字幕行是 `[时间戳 46px][gap 10][正文]`，加左右内边距约 22 →
280px 给正文留约 200px，14px 字号下约 14 个汉字一行。截图里 18 行字幕只有 4 行折行，可接受。
再窄到 260 就掉到 12 字，折行会翻倍；宽到 320 则从 ① 手里抢走 40px 而字幕并不需要。

设置列同宽的 280 顺带解决了病灶 ②：BGM 三段信息（曲名 / 标签 / 时长）在 280 里终于不打架了。

### 被否决的备选

**备选：`[预览][字幕][文案][设置]`**，让字幕直接贴着文案、预览挪到最左。
否决——把**设置和预览拉到了整个工作台的两个对角**。
调字幕高度的滑块和它影响的画面隔着两整栏，边拖边看变成来回甩头。
现状里设置紧邻预览是唯一没被用户抱怨的部分，不该顺手弄坏。

**备选：字幕列不单独成列，改成预览列下方的横向时间轴条。**
否决——18 行字幕横排要么截断要么变成横向滚动条，而字幕列表的价值恰恰是
「按时间竖着扫一遍自己的片子」（`SubtitleList.tsx` 顶部注释写得很清楚）。竖排是它的本分。

## 3. 文字平铺斜纹

### 推荐实现：**SVG `<text>` 拼成 data-URI，作为单个 `background-image` 铺在一个旋转的全屏层上**

```
背景层（#backdrop，absolute inset-0，pointer-events-none，aria-hidden）
  └─ 内层 div：inset:-60%（转 24° 后仍盖满视口）
                transform: rotate(-24deg)
                background-image: url("data:image/svg+xml,…")   ← 自动平铺
```

SVG tile 的构造（宽 = 一遍文案的长度，高 = 两行；第二行横向错开半格，避免出现竖直的接缝走廊）：

```js
function greeting () {                      // 与现有 AmbientBackdrop 的判据一致
  const h = new Date().getHours()
  return h >= 6 && h < 18 ? 'Good Noon And Good Work' : 'Good Night And Good Dream'
}

function tileUrl (text, { size = 16, tracking = 0.3, rowGap = 2.6, alpha = 0.05 } = {}) {
  const ls   = size * tracking
  const runW = Math.round(text.length * (size * 0.58 + ls))   // 0.58 ≈ 无衬线大写平均字宽
  const rowH = Math.round(size * rowGap)
  const svg =
    `<svg xmlns="http://www.w3.org/2000/svg" width="${runW}" height="${rowH * 2}">` +
    `<g font-family="Helvetica,Arial,sans-serif" font-size="${size}" ` +
    `letter-spacing="${ls.toFixed(2)}" font-weight="500" ` +
    `fill="rgb(255,255,255)" fill-opacity="${alpha}">` +
    `<text x="0" y="${Math.round(rowH * 0.72)}">${text}</text>` +
    `<text x="${Math.round(runW / 2)}" y="${Math.round(rowH * 1.72)}">${text}</text>` +
    `<text x="${Math.round(runW / 2) - runW}" y="${Math.round(rowH * 1.72)}">${text}</text>` +
    `</g></svg>`
  return `url("data:image/svg+xml,${encodeURIComponent(svg)}")`
}
```

```jsx
// ⚠️ 必须用 style property 赋值，不能拼进 style 属性字符串 —— 见下面「最容易做砸的地方」
useLayoutEffect(() => {
  const el = innerRef.current
  el.style.backgroundImage = tileUrl(greeting().toUpperCase() + '   ')
}, [greeting])
```

外层容器再加一层**外缘最实、向工作台方向消隐**的遮罩，让纹理和面板的交界不是硬切口：

```css
#backdrop {
  -webkit-mask-image: linear-gradient(90deg, rgba(0,0,0,1) 0%, rgba(0,0,0,.18) 50%, rgba(0,0,0,1) 100%);
          mask-image: linear-gradient(90deg, rgba(0,0,0,1) 0%, rgba(0,0,0,.18) 50%, rgba(0,0,0,1) 100%);
}
```

### 推荐参数（已实测，见截图）

| 参数 | 取值 | 理由 |
|---|---|---|
| 字号 | **16px** | 11px 起会碎成噪点（像 dither），22px 起变成"水印"抢戏。16 是"有质地"的甜区 |
| 字距 `letter-spacing` | **0.3em**（4.8px） | 拉开之后覆盖率降低、笔画不糊成块；也让它明显区别于正文排版 |
| 行距 | **2.6 × 字号**（≈ 42px） | 疏到能透出底色，不形成灰色块 |
| 倾角 | **-24°** | 45° 是通用的 hatch 角，读作"默认斜线纹"；-24° 读作有人排过版 |
| 不透明度 | **白 5%**（`fill-opacity 0.05`） | 见下 |
| 字重 / 大小写 | **500 / 全大写** | 大写没有升降部，斜排时上下行不互相咬 |
| 文案末尾 | 追加 **3 个空格** | 平铺时两遍文案之间要有呼吸，否则读成一长串连词 |

不透明度是唯一真正要盯的参数，实测对照：

- `assets-2026-07-19/30-tex-a-alpha045-24deg.png` — 4.5%：安静，几乎完美
- `assets-2026-07-19/30-tex-c-alpha11-24deg.png` — 11%：明显过头，开始抢注意力
- `assets-2026-07-19/30-tex-e-alpha075-24-sz22.png` — 7.5% + 22px：变成品牌水印，不是纹理

**取 5%**，配合上面那层遮罩（中段压到 18%），实际到眼睛的更低。

### 一定要做的两件收尾

- **替换、不叠加**：删掉 `AmbientBackdrop.tsx` 现有的 `repeating-linear-gradient` 45° 细线，
  以及两侧那两行竖排问候语。三种纹理叠在一起只会变成一团脏。新方案完全取代它们。
- **保留每分钟对一次时**：现有的 `setInterval(..., 60_000)` 昼夜切换逻辑照搬，
  只是重算时要重新生成 data-URI。

### 被否决的备选

**备选 A：`repeating-linear-gradient`。** 不成立——CSS 渐变画不了文字。
（现有代码用它画的是**线**，那正是用户说"不是画斜线"要换掉的东西。）

**备选 B：绝对定位的多行真实 DOM 文本**（一个转 24° 的超大 div，塞 N 行 `white-space: nowrap` 的重复文案）。
可行但否决——2560 屏上要铺满得排 60+ 行 × 每行 200+ 字符 ≈ 上万个字符节点，
每次窗口 resize 都要重新布局，而它是**装饰**，不该有这个成本。
它的唯一优势（拿到系统字体栈的中文回退）在这里完全用不上：问候语是纯 ASCII。

**备选 C：`element()` / canvas 生成贴图。** 过度工程。SVG pattern 已经是零请求、任意分辨率清晰。

## 4. 整体宽度与留白

**1200 已经不够用了，推荐提到 1440。**

算一下就清楚。四栏里有三栏是定宽的（280 + 预览 300~340 + 280 = 860~900）：

| 容器上限 | 文案栏实得 | 判断 |
|---|---|---|
| 1200（现状） | **340px** | 不可用。约 22 个汉字一行，写文案像在手机上打字 |
| 1360 | 500px | 勉强 |
| **1440（推荐）** | **540px** | 约 36 字一行，舒适；1920 屏上每边留白 240px（12.5%） |
| 1560 | 660px | 文案更宽，但留白掉到 180px（9.4%），纹理没地方铺 |

三档实测截图：
`assets-2026-07-19/43-final-1440.png`（满铺，无留白）、
`40-final-1920.png` / `60-zone-950.png`（每边 240px）、
`44-final-2560.png`（每边 560px）。

**必须知道的取舍：1440 以下的视口，斜纹背景完全看不见。**
`43-final-1440.png` 里工作台正好铺满，留白为 0。
在 1440 笔记本上、或者浏览器没最大化时，这个功能等于不存在。

**这个取舍应当接受，不要为了纹理去压窄栏宽。** 理由：纹理是装饰，栏宽是干活的地方。
把上限压到 1300 好让 1440 屏露出 70px 的留白条，那 70px 既撑不起纹理（读不出一个完整单词），
又从文案栏身上割了 140px。**用留白喂装饰是本末倒置。**
诚实的说法是：斜纹是宽屏上的一份额外质感，窄屏上工作台满铺同样成立。

**2560 上要留神**：留白涨到每边 560px，纹理面积是 1920 的 4 倍多
（`44-final-2560.png`），5% 的不透明度在那个面积上开始有点响。
若要更稳，可给背景层加一条 `@media (min-width: 2200px) { opacity: .7 }`——
但这属于调优，不是必须项。

### 栅格写法

```jsx
// 容器
<div className="relative grid h-full w-full max-w-[1440px] border-x border-line bg-ink-950
                grid-cols-[minmax(0,1fr)_280px_minmax(300px,340px)_280px]">
```

- 文案 `minmax(0,1fr)`：下限写 0 不写 520 —— 窄屏要"挤一挤"而不是顶出横向滚动条。
  这条现有注释（`Workspace.tsx:128`）的道理仍然成立，照搬。
- 预览 `minmax(300px,340px)`：预览框自身仍按 `max-w-[min(100%,26vh)]` 限宽，
  栏宽只需比它略宽即可；给上限是防它在超宽屏上喧宾夺主。
- 窄屏（< ~1180）四栏确实挤。**建议的退让顺序：先折叠设置列**（它现在有了自己的列头，
  折叠成 3.5rem 图标条是成立的，因为设置有真图标可用：`IconSubtitles` / `IconFilm` / `IconMusic` / `IconVolume`），
  字幕列最后才动——它和文案是一对。原来那套 `NARROW` matchMedia + `userDecided` ref
  的逻辑（用户手动点过就不再自动收放）可以原样复用到设置列上。

## 5. 配色系统

> 用户原话：「前端的配色和风格也要优化一下，**现在就是一片黑啊**」。
> 本节是这句话的答复。**同样没有改动 `web/src/` 下任何文件**，
> 新配色在 `assets-2026-07-19/prototype-v2.html` 里验证
> （`?p=current|A|B|C|final`、`&fx=0` 关处理、`&zone=13` 换分区、`&tex=0.045` 调纹理）。

### 5.1 先把病灶量出来

「一片黑」不是观感偏差，是可以测的。用 **CIE L\***（感知亮度）量相邻层级：

| | ink-950→900 | 900→850 | 850→800 | 800→700 | 700→600 |
|---|---|---|---|---|---|
| **ΔL\*（现状）** | **1.48** | **1.88** | **2.35** | 5.07 | 6.26 |
| WCAG 对比度 | 1.031 | 1.038 | 1.046 | 1.117 | 1.190 |

大面积平色块的**恰可察觉差（JND）约 2 ΔL\***，考虑环境光和屏幕眩光，
边界要稳定读出来需要 **≥3**。现状最关键的那一对（页面底色 vs 面板）只有 **1.48**，
**在 JND 以下**——所以「分层表面」确实只存在于命名里。

真正的坏消息还在后面。把描边也放进同一把尺子量：

```
6% 白描边叠在 ink-950 上 = #171719，L* 7.81
ink-800（第四层表面）    = #17171c，L* 7.94
```

**描边比它所分隔的三层表面都亮，且和第四层表面几乎同色。**
这就是整套东西读作「一块黑 + 几根发丝线」的机制：
**层次全是描边画出来的，表面一点忙都没帮上。** 现状截图 `70-color-before-1920.png` 里，
②③④ 三栏之间除了那根线之外没有任何边界信息。

### 5.2 新灰阶

色卡对照（含每级 L\* 与相邻 ΔL\*，红色 = 低于 JND）：**`assets-2026-07-19/72-ramp-swatches.png`**

| 令牌 | 新值 | 旧值 | L\* | ΔL\* | 为什么是这个值 |
|---|---|---|---|---|---|
| `ink-950` | `#12151b` | `#08080a` | 6.7 | — | 页面底色 / 内容区。**抬离纯黑**：纯黑在 OLED 上和描边之间没有过渡（描边直接"浮"在虚空上），在普通 IPS 上又因背光漏光读作脏灰。L\* 6.7 是"深而有底"的位置 |
| `ink-900` | `#191d23` | `#0d0d10` | 10.6 | 3.9 | 面板 / 工具区。**这一步是整个方案的关键**：3.9 已稳过 JND，2+2 分区第一次真的看得见 |
| `ink-850` | `#21252c` | `#121216` | 14.6 | 4.0 | hover 态。hover 必须一眼可辨，否则鼠标扫过没有回应感 |
| `ink-800` | `#2a2e35` | `#17171c` | 18.8 | 4.3 | 输入框 / 预览手机框。它要读作"嵌在面板里的另一种材质" |
| `ink-700` | `#363942` | `#212128` | 24.0 | 5.2 | 中性选中态 / 滑轨。步长开始放大——越往上层级越少，可以拉开 |
| `ink-600` | `#454951` | `#2e2e37` | 30.9 | 6.9 | 重分隔线 / 字标 / 版本角标 |
| `ink-400` | `#83868e` | `#6b6b78` | 56.0 | 21.0 | 次要文字。**从 L\*45.6 提到 56.0，这是一处修 bug**，见 5.8 |
| `ink-300` | `#a3a5ab` | `#9c9caa` | 67.8 | 11.8 | 说明文字。对 ink-900 达 AA |
| `ink-100` | `#dbdce0` | `#dcdce4` | 87.8 | 20.0 | 正文。几乎没动——旧值本来就对 |
| `ink-50` | `#f6f4f2` | `#f4f4f8` | 96.3 | 8.5 | 强调文字。**唯一一个转暖的令牌**，见下 |

**表面步长统一取 4 L\***（950→800 四层），不是 3 也不是 5：

- 取 3（候选 B，`75-reject-step3-1920.png`）：950→900 只有 2.4 ΔL\*，**又掉回 JND 附近**，
  分区边界在亮环境下会时有时无。既然要改，就不该改到刚好卡在阈值上。
- 取 5：ink-800 会升到 L\*23 左右，输入框和选中态挤在一起，而且整体开始不像深色工具。

四层表面总跨度从**旧的 5.7 L\*** 拉到 **12.1 L\***——翻了一倍多，但页面底色仍在 L\*6.7，
**整体依然是暗的**（`71-color-after-1920.png`）。用户要的不是"变亮"，是"看得出层"。

### 5.3 色相偏置：往冷里偏，H = 265

新灰阶不是纯灰。全部用 **OKLCH 色相 265°（蓝紫）** 生成，
彩度 **C = 0.013–0.015**（暗部）向亮部衰减到 0.003。

**为什么偏冷、且是 265 而不是青绿：**

1. **和琥珀金配得住。** `#f0b429` 的 OKLCH 色相约 **75°**，265° 正好在它对面（差 190°）。
   冷底让琥珀在**不提高自身饱和度**的前提下读得更暖更亮——这是免费的强调力度。
   反过来，暖底（候选 C）会把琥珀"吸收"进去，见 5.6。
2. **这是 SureJack 自己的行业语言。** 冷暗部 + 暖高光就是电影调色里的 teal-and-orange。
   琥珀金已经论证过它来自 ASS 字幕的 `&H0000E5FF`；冷底是同一个逻辑的另一半——
   工具的界面和它产出的片子用同一套色彩语法。
3. **为什么不是青/teal（H≈200）。** 低彩度青在偏色屏和低端 IPS 上会歪成灰绿，读作"发霉"。
   蓝紫（265）即使偏色也只是更蓝，仍然读作"墨"，容错高得多。
4. **彩度必须压到 0.015 以内。** 再高就从"带色相偏置的中性色"变成"深蓝色 UI"，
   那是另一个产品的样子。C=0.014 的效果是：单看一块认不出是蓝的，
   和纯灰并排立刻看得出偏——这正是"选过"和"没考虑过"的差别。

**唯一的例外：`ink-50` 转暖（H=80）。** 最亮的强调文字若继续偏冷，
在大片冷底上会读作发青的白，显廉价。给它一点点暖（`#f6f4f2`），
读作"暖光下的纸"，同时和琥珀同侧、遥相呼应。这是全套里唯一一次色相反向。

### 5.4 三项处理：让它不"平"

灰阶只解决了"分层"，没解决"死板"。`&fx=1` 里三件事，都极轻：

| 处理 | 参数 | 理由 |
|---|---|---|
| **上缘内高光** | `inset 0 1px 0 rgba(255,255,255,.045)` | 面板/手机框上缘 1px 提亮（ΔL\* 4.9）。**暗示光从上方来**，一条 1px 的线就够，面板因此像"抬起来的板"而不是"挖下去的洞" |
| **顶部顺光** | `linear-gradient(180deg, rgba(255,255,255,.022), transparent 220px)` | 工具区 280px×整屏的纯平面会读作死板。2.2% 的白、220px 内衰减到零，扫一眼说不出哪里不同，但那块面不再是死的 |
| **描边维持 6% / 10%** | 不变 | **现在够了**。关键不是描边变没变，是**比例变了**：旧方案描边 ΔL\*5.58 : 表面步长 1.48 = **3.8 : 1**（描边包办一切）；新方案 6.92 : 3.89 = **1.8 : 1**（表面挑大梁、描边收边）。把描边调低反而会让它在新表面上消失 |

**不建议加渐变到底色上。** 试过给页面底色加大范围径向渐变，
结果是斜纹问候语纹理和渐变互相打架，且在不同尺寸视口上渐变中心跑位。
光源暗示交给 1px 内高光就够，**装饰性渐变是最容易把"高级"做成"花哨"的一步**。

### 5.5 琥珀金：确实用得太省，但要用在对的地方

用户的怀疑是对的，而且我在现状里发现一处**层级颠倒的实际 bug**：

> `重新生成`（中间过程）= 实心琥珀按钮；`导出成片`（整条流水线的终点）= `ink-800` 幽灵按钮。
> **最重要的动作是全屏最不显眼的按钮之一。**

三处调整，**没有引入任何新颜色**，只是重新分配已有的琥珀：

1. **`导出成片` 换成实心琥珀**，`重新生成` 降级为琥珀描边的次级按钮。
   顺带把琥珀放到了右下角，和设置栏里的琥珀（背景条、选中项）形成对角平衡。
   对照见 `77-reject-export-soft.png`。
2. **选中态底色从中性 `ink-700` 换成琥珀 10% 淡染**（`rgba(240,180,41,.10)`），
   同时选中行的时间戳转琥珀。中性灰选中态读起来和 hover 分不清；
   琥珀淡染一眼就是"选中/正在播的这句"。**而且它就是这句字幕在成片里被点亮的颜色**——
   和强调色的出身论证是同一条逻辑。
3. 头像底、焦点环、左侧选中条：维持现状，本来就对。

**绝不做的事**：不引入第二个强调色；不给成功/信息状态配绿/蓝
（需要语义时用琥珀+图标，或复用已有的 `danger` 红）；`danger #e5484d` 保留不动
（对 ink-900 达 4.32，属语义色不属强调色）。

### 5.6 试过并否决的方案

**否决 A：暖调中性色 H=70（`74-reject-warm-1920.png`）**
把灰阶偏向琥珀同侧，本以为会更"统一"。实测是全套里最糟的一版：
面板读作**褐色/棕黄**，整屏像发黄的旧照片；更要命的是**琥珀被底色吸收了**——
背景条、按钮的琥珀和 `#211b16` 的面板成了近亲，强调色**丧失强调功能**。
"简洁大方高级"直接变成"复古土"。这一版最有说服力：它证明了色相偏置的方向不是随便挑的。

**否决 B：表面步长 3 L\*（`75-reject-step3-1920.png`）**
更暗、更接近现状，看着更"稳"。但 950→900 只有 **2.4 ΔL\***，
仍在 JND 边缘——**在亮环境下 2+2 分区又会消失**。
既然是专门来修"看不见"的，就不能把最关键那一对留在阈值上。

**否决 C：琥珀淡底版导出按钮（`77-reject-export-soft.png` 下半）**
`rgba(240,180,41,.13)` 底 + 琥珀描边 + 琥珀文字。想法是"实心太吵"。
实测淡琥珀叠在深底上**混成橄榄绿/土黄**，读作**禁用态**，比原来还糟。
低透明度暖色叠深底会掉彩度，这是深色 UI 的通病。终点动作就该给实心。

**否决 D：底色加大范围径向渐变**（见 5.4）。与斜纹纹理打架、随视口跑位。

### 5.7 一个必须跟着改的连带项：纹理 alpha

**底色抬亮之后，第 3 节调好的斜纹参数会失准，而且方向和直觉相反。**

| | 底色 | 纹理（白叠加） | ΔL\* |
|---|---|---|---|
| 旧 | `#08080a` | 4.5% → `#131315` | **3.71** |
| 新，沿用 4.5% | `#12151b` | 4.5% → `#1d2025` | **5.42** ← 明显变响 |
| 新，修正后 | `#12151b` | **3.2%** → | **3.58** ✓ |

直觉会以为"底色变亮了，白纹理更不明显"，**实际相反**：
alpha 合成发生在 **sRGB 编码空间**，同一个 alpha 在较亮的起点上抬升的线性光更多。
不改的话，上一轮千辛万苦调出来的"安静"会白调。

对照图 `76-tex-alpha-fix.png`（左 3.2% ✓ / 右 4.5% ✗）。
**纹理 alpha 从 4.5% 改为 3.2%。**

### 5.8 顺带修好的：ink-400 一直不达标

量的时候发现的，和本次改动无关的既有问题：

```
旧  ink-400 #6b6b78 on ink-900 #0d0d10 → 对比度 3.70   ✗ 小字不达 AA
```

次要文字（11–12px 的字数统计、时间戳、BGM 标签、hint）**一直低于 WCAG AA 的 4.5**。

**故把 ink-400 从 L\*45.6 提到 L\*56.0（`#83868e`）**，对 ink-900 达 **4.64**、对 ink-950 达 **5.02**。

### 5.10 配色上最容易做砸的地方

1. **把新灰阶当"调亮了"来理解，然后手动往回压。**
   最可能发生的事：有人觉得 `#12151b` "不够黑，没有高级感"，
   顺手改回 `#0a0a0c`——**一改就全塌**，因为整套步长是相对底色排的。
   要记住的是：**改的不是明度，是层级间距。** 页面仍然很暗（L\*6.7）。
   验收动作是**眯眼看**：应当能数出四栏里哪两栏是一组；数不出来就是改坏了。

2. **色相偏置在实现中被"清理"掉。**
   `#12151b` 看起来像"没对齐的十六进制"，很容易被顺手改成 `#121218` 或 `#131313` 求整齐。
   R/G/B 三个分量的差值**就是色相偏置本身**，抹平了就退回纯灰。
   建议在 `index.css` 注释里写死"这些值由 OKLCH H=265 生成，不要手动凑整"。

3. **纹理 alpha 忘了跟着改**（5.7）。这是最隐蔽的一条：
   纹理代码一行没动、参数一个没改，观感却变了。
   而且因为方向反直觉，排查时很可能往错误的方向调。

4. **琥珀一开闸就收不住。**
   本节只动了三处。一旦"琥珀可以多用"这个口子开了，
   很容易蔓延到小节标题、图标、滑块轨道、hover 态……
   **判据：琥珀只标记两件事——「这个是选中/正在发生的」和「这个是终点动作」。**
   不满足其一的地方一律中性色。

5. **10% 琥珀淡染在不同表面上会变色。**
   `rgba(240,180,41,.10)` 叠 ink-950 得 `#28251c`，叠 ink-900 得 `#2f2c24`——都偏橄榄。
   在字幕行（小面积、有琥珀时间戳和左条锚定）成立；
   **若有人把它搬去做大面积容器底，就会变成 5.6 里否决的那摊土黄。**
   这个令牌只给行级选中态用。

6. **别在 OLED 上定底色。** ink-950 抬离纯黑的收益（描边有过渡、不脏）
   在 OLED 上看不出来——OLED 上纯黑反而"更帅"。
   但用户不一定用 OLED，而纯黑在 IPS 上的问题是真的。**在普通 IPS 上定这个值。**

## 6. 最容易做砸的地方

按「踩中概率 × 修起来有多烦」排序：

1. **data-URI 拼进 `style` 属性字符串会静默失效。**
   本轮原型就是这么栽的一次：`url("data:…")` 里的双引号把 HTML 属性提前闭合了，
   结果 `getComputedStyle` 拿到的是 `url("")`——**没有报错、没有警告，就是一片纯黑**，
   看起来完全像"纹理太淡了"，于是会去调 alpha，越调越糊涂。
   **必须 `el.style.backgroundImage = …` 走 property 赋值**，或者把 `url()` 里改用单引号。
   建议顺手写一条断言：`getComputedStyle(el).backgroundImage.length > 100`。

2. **纹理不透明度会被"看着调"调过头。**
   写代码时人是盯着留白区看的，5% 会觉得"太淡了，看不清"。
   但它的正确用法是**眼睛看着右边写字时的余光**。
   验收动作要固定成：光标放进文案框、视线落在正文上，再问余光里那片东西烦不烦。
   宁淡勿浓。别在 2560 的显示器上定这个值（面积 4 倍，判断会偏淡）。

3. **①② 两列共用 `ink-950` 这件事会在实现中悄悄丢掉。**
   四个 `<section>` 长得一样，很容易顺手全写成 `bg-ink-900`。
   一旦丢掉，四栏就摊成四个等重的窗口，「文案和字幕是一回事」这个信息随之消失
   ——而这正是用户对互换方案的核心顾虑。对比 `60-zone-950.png` 与 `40-final-1920.png` 能看出差别有多大。

4. **项目切换器的 popover 层级和关闭行为。**
   它挂在列头里，父级链上有 `overflow-hidden` 的话会被裁掉。
   还要处理：点外部关闭、`Esc` 关闭、打开时按钮保持"按下"底色（否则鼠标一移开按钮就恢复常态，
   浮层像是凭空飘着的）。删除项目的 `confirm()` 弹出时不能顺带把 popover 关了。

5. **别在全局 `:focus-visible` 里加 `border-color`。**
   已经踩过的坑（`index.css:66-79` 有完整记录）。
