# 背景轨提前拼好 + 预览显示背景 实施计划

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development to implement this plan task-by-task.

**Goal:** 配音一就绪就在后台把背景轨拼好，导出时直接用，预览里也能看到真实背景。

**Architecture:** 素材已归一化成同一规格，所以切片可以用 `-c copy` 直接拷贝压缩帧——**实测 88 秒 → 0.24 秒，366 倍**。拼接因此从"半小时的大工程"变成"十几秒的后台小活"，可以在配音完成后立刻做掉。

**Tech Stack:** TypeScript / ffmpeg / Fastify / React 19 / vitest

## Global Constraints

- 测试框架 **vitest**；`npm test` 先跑 `tsc --noEmit && tsc -b web`
- **每个 `expect(...)` 必须接 matcher**，否则假绿
- noUncheckedIndexedAccess 开着，**不要用 `!` 绕过**
- **不要用 `as XxxType` 硬转造测试数据**（踩过：`fitMode` 误写成 `fit`，跑到 ffmpeg 才炸）
- **时长必须整数毫秒**（踩过：Azure 的 HNS 除以 10000 出小数，背景排布直接 500）
- **不要用 emoji**；不引入新颜色

## 关键实测数据

同一个已归一化的跑酷素材，截 10 秒：

| 做法 | 耗时 | 得到的时长 |
|---|---|---|
| 重新编码（当前实现） | 88.03s | 10.000s |
| `-c copy` 直接拷贝 | **0.24s** | 10.100s |

拷贝快 366 倍，代价是切点吸附到最近的关键帧（归一化时设了 `-g 30`，即每秒一个）。

**这个误差无所谓，但必须理解为什么无所谓：**
1. 背景是填充画面，段边界差 0.1 秒没有任何人看得出
2. 总长由烧录那步的 `-t durationSec` 精确截断
3. 所以只要保证**拼出来的总长 ≥ 配音时长**即可——每段向上取整，宁长勿短

**短了会怎样**：`buildArgs` 对背景加了 `-stream_loop -1`，短了会从头循环，接缝处画面突然跳回开头，很明显。所以宁可长。

### Task 1: 切片改用拷贝

**Files:** `src/compose/build.ts`、测试

- 源文件用 `resolveSource()`（`src/library/normalize.ts`）取，**已归一化的用归一化版，没转过的退回原文件**
- 归一化过的：`-ss X -i src -t Y -c copy -an`
- **没归一化的仍走重编码**——原文件规格杂乱，拷贝拼不起来（concat demuxer 要求参数一致）
- 每段向上取整到关键帧，保证总长 ≥ 需求

**测试必须覆盖：**
- 拷贝路径产出的总长 ≥ 请求时长（不能短）
- 混合情况（部分素材已归一化、部分没有）能正确拼接
- 拷贝路径的产物无音轨、规格与目标一致

**这条最关键**：拷贝路径若把规格不一致的文件拼在一起，ffmpeg 不一定报错，可能产出一个**只有第一段能正常播放**的文件。要有测试真的解码后几段来验证，不能只看时长。

### Task 2: 配音就绪后自动拼背景轨

**Files:** `src/queue/`（复用现有队列）、`src/tts/routes.ts`、`src/projects/routes.ts`、测试

配音完成后，把"拼背景轨"作为一个后台任务入队。

**要点：**
- **复用现有的串行队列**，不要另起一套并发。机器 4 核，导出本身就吃满 CPU
- 产物落 `assets/<projectId>/bg-track.mp4`，登记成 `kind='bgtrack'` 的素材（或project 上加一列记录路径，实现者定，在报告里说明）
- **排布变了要重拼**：素材库扫进新文件会改变排布（已知取舍，见 `background.ts` 注释）。存一个排布指纹（segments 的 itemId+takeMs 拼起来哈希），不一致就重拼
- 导出时若背景轨已存在且指纹匹配，**直接用，跳过生成**
- **失败不能阻断导出**：后台拼失败时，导出回退到现在的即时生成路径。用户不该因为一个优化没成功就导不出片子

---

### Task 3: 预览显示背景

**Files:** `web/src/components/Preview.tsx`、测试

背景轨拼好之后，预览直接播它——`<video>` 的 src 指向 `/api/assets/<bgtrack id>`。

**这比"浏览器按排布逐段播"简单得多**，因为轨已经是一整条了。现有的"画面跟随音频、漂出 150ms 才校正"那套逻辑一行都不用改。

**状态要说清楚：**
- 背景轨还在后台拼 → 显示「背景生成中…」，别让用户以为坏了
- 拼好了 → 正常播
- 拼失败 → 显示「预览暂无背景，导出时会重新生成」，**不要让预览的失败看起来像导出会失败**

## 磁盘账

背景轨约 **65MB/分钟**（实测 65 秒的 track 是 69MB）。一条 11.5 分钟的片子约 750MB。

**这是真实成本**，要在计划里认账：
- 3 条片子 ≈ 2.2GB
- 当前可用约 30GB

**需要清理策略**：导出完成后背景轨是否保留？保留则预览随时可看，但磁盘线性增长。建议**保留，但在项目删除时一并清掉**，并在设计文档里记下这个增长点。
