# SureJack 设计文档

> 日期：2026-07-16
> 状态：待评审

## 1. 这是什么

一个特化的营销号视频生成工具。你给它两样东西——**一段文案**和**一个背景视频**——它生成一条可以直接发布的短视频：AI 配音、自动字幕、固定标题和免责声明，全部烧录在画面上。

它不是剪辑软件。它不追求通用，它只把"营销号视频"这一件事做到最省事。

**规模**：最多两人同时使用。这不是一句免责声明，而是一个**设计许可**——它让我们可以合理地拒绝分布式队列、对象存储、真实认证这一整类复杂度。

## 2. 核心决策速查

| 决策 | 选择 | 理由 |
|---|---|---|
| 配音 | Azure TTS 免费层（F0） | 每月 50 万字符、按月重置、超额只限流不扣费 |
| 字幕时间 | 完全由配音的字级时间戳推导 | 只有一个真相来源，永远不会和配音脱节 |
| 断句 | 全自动（标点 + 字数上限） | 文案是自己写的，想断哪儿加个标点即可 |
| 字幕与文本层 | 统一为一个 ASS 文件 | 同一个 libass 渲染预览和成片，所见即所得由架构保证 |
| 预览渲染 | JASSUB（libass 的 WebAssembly 版） | 与导出用的是同一个 C 库，像素级一致 |
| 合成 | 服务端 ffmpeg，`child_process.spawn` 直调 | `fluent-ffmpeg` 已归档废弃且与新版 ffmpeg 不兼容 |
| 前端 | Vite + React + TypeScript | 生态最深，JASSUB / 拖拽裁切这类底层活儿现成答案多 |
| 后端 | Node 24 LTS + Fastify + TypeScript | Node 20 已于 2026-04-30 EOL |
| 数据库 | SQLite（`better-sqlite3`），**每个用户一个独立文件** | 隔离是物理性的，跨用户泄露在结构上不可能发生 |
| 登录 | **真实认证**：白名单 + 密码 + HTTPS + 限流 | 服务在公网上，不是玩笑 |
| 导出队列 | 进程内队列 | 两个用户上分布式队列纯属自残 |

## 3. 用户与登录

**登录是真的安全边界。** 这个服务跑在公网上（`surejack.zacchen.win`），所以本节每一条都不是可选项。

### 流程

- **白名单两个姓名**，硬编码在配置里，不入库
- **名单内、首次登录** —— 设置密码并保存
- **名单内、已设密码** —— 验证密码后进入，各自看到**专属欢迎页**
- **名单外** —— 拒绝登录，退回初始页

> 名单和欢迎页的具体内容**留待实现时确定**，不阻塞架构。
>
> "名单外直接拒绝"这个交互天然会泄露白名单的存在（能从响应区分"名字不在名单"和"密码错误"）。这是可接受的——**登录名本来就是真实姓名，从来不是秘密**，把它当秘密来保护是自欺欺人。密码才是那道门。

### 已知风险：抢注（已评估并接受）

公网上，**任何知道白名单姓名的人都可以抢先设走密码**。真实姓名不是秘密，所以这不是理论风险。

**但它被接受了，理由成立**：
- **损害小** —— 抢注发生时账号里没有任何数据，攻击者只是占了个坑
- **可检测** —— 本该看到"设置密码"却看到"输入密码"，信号非常明确
- **可恢复** —— 管理员有服务器权限，重置即可

**因此产生三条硬性要求**：

1. **必须实现密码重置 CLI**。"被抢了就在后端重置"这个兜底要成立，这条命令必须真实存在。**而且它本来就是必需品**——没有邮箱验证、没有找回流程，忘记密码的唯一解就是它。
2. **首次设密码必须记日志**（时间 + 来源 IP）。这是判断"是否被抢注、被谁抢"的唯一依据。
3. **部署后立刻设置两人的密码**，把攻击窗口关掉。这是上线检查项，不是建议。

### 安全实现要求

| 项 | 要求 |
|---|---|
| 密码存储 | `argon2id`，或 `node:crypto` 内置的 `scrypt`（无原生依赖）。**绝不明文、绝不 MD5/SHA** |
| 登录限流 | 必须。密码是唯一的门，不限流等于允许慢速爆破 |
| 会话 | 签名 cookie，`httpOnly` + `secure` + `sameSite=lax` |
| 传输 | HTTPS 强制，HTTP 跳转到 HTTPS。见第 16 节 |
| 上传 | 文件类型校验、大小上限、防路径穿越 |

### 数据隔离：每人一个独立数据库

```
data/
├── <姓名A>/
│   ├── app.db          ← 只有 A 登录后才会打开
│   └── assets/
└── <姓名B>/
    ├── app.db
    └── assets/
```

**隔离是物理性的，不是逻辑性的。** 打开哪个数据库文件由会话身份决定，代码里**不存在** `WHERE owner = ?` 这种东西——因为不需要，打开的就是你的库。这意味着"某处查询忘了加过滤条件"这类数据泄露**在结构上不可能发生**，而不是"靠仔细避免"。

**素材文件同理**：路径由会话身份拼出，**绝不接受 URL 参数指定路径**。所以"改个 URL 看别人的视频"这条路不存在。

代价是没法跨用户查询、schema 变更要对每个库各跑一次。两个用户的规模下，这些代价约等于零，换来的结构性安全非常划算。

## 4. 数据模型

**两层数据库。**

**`data/auth.db`（根级，唯一一个共享库）** —— 只管认证，不含任何项目数据
- `name`（白名单内的姓名，主键）, `password_hash`
- `created_at`, `first_login_ip` —— 抢注检测的依据，见第 3 节

> 认证数据独立成库，是因为它是**横切关注点**，不属于任何一个用户的项目数据。密码重置 CLI 只碰这一个文件。

**`data/<姓名>/app.db`（每人一个）** —— 以下所有表都在这里。**没有 `owner` 字段，因为整个文件就是一个人的。**

表结构如下（省略了显而易见的时间戳字段）。

**`projects`** —— 项目，核心是文案
- `id`, `name`
- `script_text` —— 文案正文，项目的一等公民
- `script_hash` —— 文案 + 配音参数的哈希，用来判断配音是否过期
- `aspect_ratio` —— `9:16` / `4:5` / `1:1` / `16:9`
- `voice_name`, `voice_rate`, `voice_volume` —— 配音参数（`voice_volume` 是 SSML 层面的合成音量）
- `subtitle_mode` —— `line`（整行显示）/ `karaoke`（逐字高亮）
- `subtitle_style_json` —— 字号、颜色、描边、垂直位置
- `subtitle_max_chars` —— 单行字数上限，默认 15，超出强制拆行
- `bgm_volume` —— BGM 相对配音的混音音量，即"音量平衡"滑杆
- `tts_state` —— `none` / `generating` / `ready` / `stale` / `error`
- `tts_duration_ms`, `word_timings_json` —— 配音结果

**`assets`** —— 一切上传和生成的文件
- `id`, `project_id`, `kind`（`video` / `bgm` / `voice` / `export` / `source_doc`）
- `path`, `original_name`, `mime`, `size`
- `duration_ms`, `width`, `height` —— 由 ffprobe 探测填入

**`clips`** —— 背景视频片段（有序）
- `id`, `project_id`, `asset_id`, `order_index`
- `fit_mode` —— `cover`（裁切填满）/ `contain`（留黑边）/ `blur`（模糊填充）
- `crop_offset_x`, `crop_offset_y` —— **裁切窗口中心点在源画面中的归一化位置**，0..1，默认 0.5 / 0.5（居中）。仅 `cover` 模式有意义。这个定义必须在前端预览和 ffmpeg 命令两处严格一致，见第 15 节风险 5
- `trim_start_ms`, `trim_end_ms` —— 片段自身的裁剪，可空

**`text_overlays`** —— 固定位置文本
- `id`, `project_id`, `content`, `preset`（`title` / `disclaimer` / `custom`）
- `pos_x`, `pos_y` —— 归一化坐标，对应自由拖动
- `align` —— 九宫格对齐，对应预设档位
- `start_ms`, `end_ms` —— 可空，空 = 全程常驻
- `style_json`, `order_index`

**`export_jobs`**
- `id`, `project_id`, `status`, `progress`, `error`, `output_asset_id`

> **白名单本身不入库**，它在配置文件里。姓名 → 数据库路径的映射**必须先经过白名单校验**再拼路径——这是防路径穿越的第一道闸，不能省。

**刻意不存的东西：字幕行。** 字幕行是**推导数据**——由 `word_timings_json` 加断句规则算出来的。存它就等于制造第二个真相来源，就等于制造"文案改了但字幕没跟上"这类 bug。它每次都算，算起来不要钱。

## 5. 文案导入

四个入口，**粘贴是主路径**，文件上传是辅助：

| 格式 | 方案 | 风险 |
|---|---|---|
| 直接粘贴 | 无需处理 | 无 |
| `.txt` | `chardet` 探测编码 + `iconv-lite` 转 UTF-8 | **中文 txt 大量是 GBK/GB18030**，按 UTF-8 硬读必然乱码 |
| `.docx` | `mammoth` 提取纯文本 | 低 |
| `.doc` | **`catdoc`**（已实测；`antiword` 对中文直接崩溃，出局） | **catdoc 静默失败——退出码不可信，见下** |

`.doc` 做成**尽力而为**：抠得出来就用，失败或乱码就明确告诉用户"请在 Word 里另存为 .docx 再传"，绝不假装成功让乱码进入配音环节。

> **⚠️ catdoc 会静默失败（已实测，见 `docs/superpowers/spikes/RESULTS.md`）**：喂它一个非 `.doc` 文件，它**吐出乱码却返回退出码 0**。所以**绝不能只看退出码**——必须做基于内容的校验：抽取结果中若 CJK 码点占比过低、而 Latin-1 补充区（`À`–`ÿ`）字符占比异常高，判定为乱码并拒绝。
>
> 另：catdoc **忽略 `-s` 参数**，只认文档头声明的 code page。且「真实 cp936 中文 Word 2003 文件能否正确解析」**尚未验证**（无法用 LibreOffice 造出可信样本），拿到真实样本后应复验。基于内容的校验正是为这个不确定性兜底的。

**必须处理 `<`、`>`、`&`。** 但原因和最初设想的不同——**已实测（SDK 1.50.0 / zh-CN-XiaoxiaoNeural），见 `docs/superpowers/spikes/RESULTS.md`**：

> 早前依据 [issue #2359](https://github.com/Azure-Samples/cognitive-services-speech-sdk/issues/2359) 断言「特殊字符会导致其后所有时间戳错乱」——**这条没有复现**，时间戳完全正常、单调递增。该 issue 要么已修，要么只在特定音色/条件下触发。
>
> **真实的问题是 XML 实体转义**：输入 `A&B`，`WordBoundary` 事件回来的 `text` 是 **`&amp;`**（5 个字符），不是 `&`。SDK 把文本包进 SSML 时做了转义，而事件报告的是转义后的形态。两个后果：
> 1. **字幕会字面显示 `&amp;`** —— 用户看到实体码
> 2. **`textOffset` 指向 SSML 字符串的位置，不是原文** —— `&`→`&amp;` 长度由 1 变 5，其后所有偏移量错位
>
> **对策**：`subtitles/` 在用 `e.text` 构造字幕行时**必须反转义 XML 实体**（`&amp;`→`&`、`&lt;`→`<`、`&gt;`→`>`）。若需要用 `textOffset` 回溯原文，必须先补偿转义带来的长度差——**或者干脆不依赖 `textOffset`，只用 `e.text`**（推荐，少一个出错来源）。

## 6. 配音

**用 SDK，不用 REST。** Azure 的 REST TTS 接口只吐音频，**字级时间戳只通过 WebSocket 的 `WordBoundary` 事件推送**，只有官方 SDK 拿得到。这条不能省。

**整篇一次合成，绝不逐句请求。** F0 的限速是 **每 60 秒 20 次请求**。一篇 30 句的文案逐句合成就是 30 次请求，直接撞墙。单次请求最长可出 10 分钟音频，营销号绰绰有余。超过 10 分钟的文案直接拒绝并说明原因。

**`WordBoundary` 事件字段**：
- `audioOffset` —— 单位是 100 纳秒，**除以 10000 得毫秒**
- `duration`, `text`, `textOffset`, `wordLength`
- `boundaryType` —— `Word` / `Punctuation` / `Sentence`

**标点会单独触发事件**（实测 `boundaryType` 取值为 `WordBoundary` / `PunctuationBoundary`），所以中文断句是白送的——不需要自己写分词。中文没有空格的问题由 Azure 内部解决。

> **⚠️ 时间戳粒度是「词」不是「字」**（已实测）：Azure 自己做了中文分词，「震惊」「知道」「什么」都是单个整词事件。因此 **`\kf` 标签必须按词分组生成，不能按字**——卡拉OK 的效果是「震惊」两个字一起亮。这比逐字更自然，真正的卡拉OK 本来就是按词走的。
>
> 数字和英文边界很干净：`99%`、`AI` 各自是一个完整事件，不会被拆碎。

**配音是手动触发的，不是自动的。** 改一个字就重新配音会烧配额、还会撞限速。流程是：写文案 → 点"生成配音" → 拿到音频和时间戳 → 预览。之后改文案，配音标记为 `stale`，界面提示需要重新生成。

## 7. 字幕与文本层

**它们是同一个东西。** 字幕、顶部标题、底部免责声明，全部是一个 ASS 文件里的事件行，只是 Style 和 Layer 不同。

ASS 原生提供了我们需要的一切：
- `\pos` —— 自由拖动定位
- `\an` 九宫格对齐 —— 预设档位
- 每行自带起止时间 —— 常驻就是 `0` 到片尾
- `\kf` —— **卡拉OK 逐字高亮，libass 原生渲染，我们不用画任何一帧**

**断句规则**（全自动）：按标点边界断行，叠加一个字数上限强制拆分，避免竖屏放不下。每行的起止时间 = 该行首字的 `audioOffset` 到末字的 `audioOffset + duration`。

**换行由我们写死。** libass 有[中文自动换行的已知问题](https://github.com/libass/libass/issues/715)——但我们本来就自己控制每行字数，会在 ASS 里显式写换行符，根本不让 libass 去猜。

**`PlayResX` / `PlayResY` 必须等于输出分辨率**，否则预览和成片的字幕大小会漂移。

**字体：思源黑体**，开源、免费商用、够粗压得住画面。

> **ASS 里的 `Fontname` 必须精确写 `Noto Sans CJK SC`**（已实测确认，见 `docs/superpowers/spikes/RESULTS.md`）。**不是 `Noto Sans SC`**——那个族名不存在，`fc-match` 会**静默回退**到 DejaVu Sans（零个中文字形），表现为字幕渲染成方块或完全不显示，**且 ffmpeg 不报错**。
>
> 这个值必须是**配置常量，不是散落的字符串字面量**，且服务启动时应校验字体可解析——静默失败的东西必须主动探测。
>
> 字体文件：`/usr/share/fonts/opentype/noto/NotoSansCJK-Bold.ttc`（20.0 MB）。

> **注意**：JASSUB **不能使用系统字体**，必须由前端显式提供字体文件。所以同一个字体文件要同时给到两端——服务器上给 ffmpeg（`fontsdir`），HTTP 给前端（JASSUB）。完整 CJK 字体十几 MB，首次加载有感知，需要缓存。**两端必须是同一个文件**，否则"同一个渲染器"的保证就破了。

## 8. 背景视频与画幅

**画幅是项目级设置**，四个预设：`9:16` / `4:5` / `1:1` / `16:9`。

**每个片段独立决定怎么塞进这个画幅**：
- `cover` —— 缩放填满 + 裁切，**可拖动调整取景**（默认）
- `contain` —— 完整保留 + 黑边
- `blur` —— 居中 + 同画面放大模糊铺底

**时长逻辑。** 设配音时长为 `T`，片段按顺序拼接后的总长为 `L`：
- `L > T` —— 截断到 `T`
- `L < T` —— 整个序列从头循环，直到 `T`

> **这里和早前的说法有出入，需要确认。** 讨论时曾把"拼接 / 循环 / 截断"定为三个可选的**填充策略**。写文档时发现它们其实不是三个互斥的模式，而是**同一套规则在不同情况下的自然结果**：拼接就是片段数大于一，循环和截断是 `L` 与 `T` 不匹配时唯一合理的行为。所以这里**没有给用户开关**，也没有 `fill_mode` 字段——一套规则覆盖三种说法，不需要写三份代码。
>
> 如果确实想要一个显式开关（比如"我不要循环，就让画面定格在最后一帧"），说一声就加，代价很小。但在没有具体场景之前不做。

**背景视频一律静音。** 原声不保留。

## 9. 音频

三条轨，规则简单：
- **配音** —— 主轨，决定总时长
- **BGM** —— 可选，单独上传，一个音量平衡滑杆。不足 `T` 则循环，超出则截断
- **背景视频原声** —— 丢弃

**不做侧链压缩**（BGM 在说话时自动压低）。用户明确说了不要这个复杂度，音量平衡滑杆够用。

## 10. 导出流水线

**单片段走一趟，多片段走两趟。** 这不是优化，是必要——ffmpeg 的 `loop` 滤镜按帧工作、吃内存，而 `-stream_loop` 只能作用于**输入文件**，没法作用于拼接结果。所以：

**快路径（单片段，最常见）**：一趟 ffmpeg。`-stream_loop -1` 直接循环输入 → 归一化到画幅 → `-t T` 截断 → 烧 ASS → 混音 → 输出。

**慢路径（多片段）**：两趟。第一趟把各片段归一化到统一画幅并拼接，输出无音频的中间文件（`-crf 16 -preset veryfast`，画质损失可忽略）；第二趟 `-stream_loop -1` 循环中间文件 → `-t T` → 烧 ASS → 混音 → 输出。

**输出**：H.264（`libx264`）+ `yuv420p` + AAC。`yuv420p` 不是可选项——不加它，成片在部分播放器和平台上直接不能播。

**进度**：`ffmpeg -progress pipe:1` 输出 `out_time_ms`，除以 `T` 得百分比，经 **SSE** 推给前端。

> **SSE**（Server-Sent Events）是一个让服务器单向持续给浏览器推消息的标准机制。用它而不用 WebSocket，是因为进度只需要**服务器往浏览器推**，不需要反向通信——SSE 是这个场景的原生答案，比 WebSocket 简单一个数量级。

## 11. 前端架构

Vite + React + TypeScript。

- **状态管理：Zustand** —— 编辑器有大量客户端状态（当前项目、播放位置、拖拽中的取景偏移）。Zustand 是个轻量状态库，比 Redux 简单得多，适合这个规模。
- **UI：Tailwind + shadcn/ui** —— 侧边栏折叠、滑杆、弹窗都是现成的，不重造。
- **预览**：`<video>` 元素叠 JASSUB。JASSUB 用 WebGL 渲染 ASS，性能足够实时。
- **时间轴和裁切拖拽**：自己写。都是几百行的事，引库反而受制于人。

**布局**：左侧可折叠项目列表 → 中间预览 + 时间轴 → 右侧属性面板（文案、配音参数、字幕样式、文本层）。

### 界面风格（硬性要求）

**简洁大方、配色高级、有设计感。** 这是明确的产品要求，和功能同等重要，不是做完功能之后再"美化"一下。

**深色为主。** 这不是审美偏好，是品类的功能性选择——专业视频工具（Premiere、DaVinci、剪映）无一例外是深色的，因为**界面必须退后，让视频画面成为唯一的视觉焦点**。浅色界面会和视频内容抢注意力，长时间盯着也累。

**配色克制。** 一个中性灰阶做骨架，**一个**强调色用于交互焦点和状态。不堆颜色。"高级"在实践中基本等于"克制"——少量颜色、充足留白、清晰的层级。

**排版是主要的设计手段。** 靠字重和字号建立信息层级，而不是靠框线、分割线和色块把界面切得七零八落。

**动效有目的**：状态转换、进度反馈、拖拽跟手。不做装饰性动画。

> **审美这件事写在文档里是不够的。** 实现时会**先做一版界面骨架给用户看**，确认方向对了再往里填功能——比全做完再返工便宜得多。上面这些是设计约束，不是最终答案。

**组件边界原则**：预览、时间轴、文案编辑器、属性面板各自独立，只通过 store 通信。任何一个都应该能单独理解和测试。文件变大就是拆分的信号。

## 12. 后端架构

Node 24 LTS + Fastify + TypeScript。模块划分如下，每个模块一个明确职责：

- **`importers/`** —— 文案导入。输入文件，输出干净的 UTF-8 文本。编码探测、格式解析、特殊字符清洗都在这里，外面不需要知道文件格式的存在。
- **`tts/`** —— 封装 Azure SDK。输入文本 + 参数，输出音频文件 + 字级时间戳。**这是一个可替换接口**——哪天换服务商，只动这一个模块。
- **`subtitles/`** —— 纯函数。输入字级时间戳 + 断句规则 + 样式 + 文本层，输出 ASS 字符串。**没有 IO，最好测。**
- **`render/`** —— 构造 ffmpeg 命令、执行、解析进度。
- **`queue/`** —— 进程内作业队列，串行执行导出。
- **`auth/`** —— 白名单校验、密码哈希与验证、会话、登录限流。**姓名 → 数据库路径的映射只在这里发生**，是防路径穿越的唯一入口。
- **`db/`** —— SQLite 访问。打开哪个库由会话身份决定，模块外部无法指定路径。
- **`cli/reset-password`** —— 独立的命令行工具，重置指定姓名的密码。**这是必需品**：忘记密码的唯一解，也是抢注风险被接受的前提（见第 3 节）。

## 13. 错误处理

对每一类失败，都要给出**用户能据此行动的**信息，而不是"出错了"：

| 失败 | 处理 |
|---|---|
| Azure 限流（429） | 指数退避重试。文档明确要求实现重试逻辑 |
| Azure 月度配额耗尽 | 明确告知"本月免费额度已用完，下月 1 号重置"，不重试 |
| 文案超 10 分钟音频 | 提交前按字数估算并拦截，说明 F0 单次上限 |
| `.doc` 解析失败/乱码 | 提示另存为 `.docx`，不吞掉 |
| `.txt` 编码探测置信度低 | 提示可能乱码，让用户确认或改用粘贴 |
| 视频文件无法解码 | 上传后立刻 ffprobe 探测，坏文件当场拒绝，不拖到导出时才炸 |
| ffmpeg 失败 | 捕获 stderr 完整记录，界面给出可读摘要 |
| 磁盘写满 | 导出前检查可用空间 |
| 密码错误 | 提示错误，计入限流。**不区分**"密码错"和"还没设密码"以外的细节 |
| 登录触发限流 | 明确告知需要等待多久，不静默失败 |

**原则：早失败。** 所有能在上传/提交时发现的问题，绝不留到导出时——用户等了 40 秒才看到报错，是最糟糕的体验。

## 14. 测试策略

- **单元测试**（主战场）：断句逻辑（时间戳 → ASS 行）、编码探测、ASS 生成、ffmpeg 命令构造、时长计算（`L` 与 `T` 的各种组合）。这些都是纯函数，覆盖率应该很高。
- **集成测试**：TTS 打桩；ffmpeg 用几秒的小样本真跑，验证成片时长、分辨率、音轨存在。
- **端到端**：一条完整链路——粘贴文案 → 生成配音 → 导出 → 检查成片。跑得慢，只保一条。

## 15. 风险与早期验证

**这些必须在动手写功能之前验证掉。** 每一条如果不成立，架构都要改——等写完再发现就是灾难。

1. **卡拉OK 标签**：`\kf` 在 ffmpeg/libass 烧录和 JASSUB 预览里都正常渲染，且效果一致。libass 宣称完整支持 ASS，JASSUB 宣称支持全部特性，**但我没有实测过**。这是整个"两端同一渲染器"架构的地基，必须第一个验。
2. **中文 `.doc`**：拿真实的中文老 `.doc` 文件实测 `antiword` 和 `catdoc`，确定哪个能用、还是都不能用。
3. **Azure zh-CN 字级时间戳**：注册 F0 后实测中文 `WordBoundary`，确认时间戳精度和标点事件行为符合预期。
4. **JASSUB 字体加载**：十几 MB 的 CJK 字体在前端的实际加载体验。
5. **ffmpeg 滤镜链**：裁切偏移的坐标系在预览和成片之间必须精确一致。这是最容易出现"差一点"的地方。
6. **443 入站可达性**：DNS 已确认正确解析到本机，80 端口已确认可达（`plus` 站在跑），**但 443 目前无人监听，能否从公网连通完全未经验证**。防火墙或云安全组挡住 443 是常见情况。**这条不通，HTTPS 就没了，整个"公网 + 真实认证"的前提当场崩塌**——所以它必须和第 1 条一样，排在写任何功能之前。

## 16. 部署

**域名与网络**（已确认可用）：

```
surejack.zacchen.win  →  130.245.136.191   ← 本机出口 IP，Cloudflare DNS only（灰云）
```

**必须是 DNS only，不能开橙色云代理。** [Cloudflare 免费版代理有 100MB 的请求体上限](https://developers.cloudflare.com/workers/platform/limits/)，而背景视频随便一条 1080p 素材就能超过——开代理等于当场废掉核心功能。代价是服务器真实 IP 暴露、HTTPS 要自己配，这个交换在本项目里明确划算。

> ⚠️ **这台机器上已有生产服务**：`plus.drziangchen.uk`（nginx `:80` → `127.0.0.1:8808`），走 Cloudflare 橙色云代理，所以它没有本地证书、只听 80。**SureJack 必须用独立的 server block 和独立端口，配 certbot 时绝不能动它的配置。**

**前后端同域，由同一个 nginx 提供。** 前端是静态文件（Vite 打包产物），后端是 API。**不拆到 GitHub Pages**——那会让 cookie 变成跨站，把登录搞复杂，而且换不来任何好处：服务器无论如何都得开着跑 ffmpeg，让它顺手发几个静态文件成本为零。同源之后 cookie 自动就对，CORS 根本不存在。

**需要安装**：ffmpeg（含 `--enable-libass`）、Node 24（当前是已 EOL 的 20）、`fonts-noto-cjk`、`antiword`/`catdoc`、`certbot`

**上线步骤**（顺序有依赖）：
1. **验证 443 入站可达** —— 目前无人监听 443，能否连通未经验证。80 已确认可达（`plus` 站在跑）。**这条不通，后面全都免谈**
2. certbot 签发证书（Let's Encrypt）
3. nginx 新增 server block，HTTP 强制跳 HTTPS
4. systemd 起后端服务
5. **立刻设置两人的密码**，关闭抢注窗口（见第 3 节）

**备份**：打包 `data/` 目录即可——里面是 `auth.db`、每人的 `app.db` 和素材。

**可选：GitHub Actions 自动部署。** push 代码 → 自动编译前端 → SSH 到服务器更新并重启。这是 Actions 的正确用法（临时工人来干活），不是拿它当服务器——Actions 的机器是临时的、无对外地址、跑完即销毁，托管不了任何东西。

## 17. 明确不做

YAGNI。以下每一项都被考虑过并**主动拒绝**，理由记录在此，免得日后重新争论：

- **字幕手动调时间** —— 会引入缩放、撤销栈、冲突处理，以及"改文案后手调数据怎么办"这个无解矛盾。时间永远是配音的函数。
- **字幕手动断句** —— 想断哪儿在文案里加标点即可。断句逻辑是独立函数，日后要加覆盖层不用返工。
- **侧链压缩** —— 用户明确拒绝。
- **邀请码 / 命令行预设密码** —— 曾提议用来堵住抢注（见第 3 节）。被拒绝，理由成立：抢注损害小、可检测、可恢复，不值得牺牲自助设密码的流程。**但重置 CLI 仍然必须实现**——它是这个决定成立的前提，也是忘记密码的唯一解。
- **邮箱验证 / 密码找回流程** —— 两个用户，重置 CLI 就是全部答案。
- **分布式队列 / 对象存储** —— 两个用户。
- **转场特效 / 素材库 / 字体上传** —— 不在需求内。字体上传还额外带来版权问题。
- **浏览器内视频编码（WebCodecs）** —— 兼容性、内存、编码质量不可控。"先进"应该体现在架构上，不是把编码硬塞进浏览器。
