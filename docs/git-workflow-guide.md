# Git 分支与团队协作完全指南

> 为初学者写的、从零开始的 Git 工作流指南。
> 每个概念都配一个**我们在 SureJack 项目里的真实操作**作为例子。
> 读完你应该能看懂企业里同事说的「拉个分支」「提个 PR」「rebase 一下」到底在干什么。

---

## 0. 先建立一个心智模型

在学任何命令之前，先理解 Git 到底是什么。

**Git 是一台「时光机 + 平行宇宙」机器。**

- **时光机**：它记录你代码的每一个快照（叫「提交 / commit」），你随时可以回到任何一个历史节点。
- **平行宇宙**：它允许你从某个时间点「分叉」出一条独立的时间线（叫「分支 / branch」），在里面折腾，不影响主时间线。折腾好了，再把两条时间线「合并 / merge」回去。

企业里所有的协作流程，本质都是在管理这些平行宇宙：**谁在哪条时间线上干活、什么时候合回主线、合之前谁来检查**。

## 1. 四个核心概念

### 1.1 提交（Commit）—— 一个存档点

一次 commit 就是给你当前的代码拍一张快照，存下来。每个 commit 有：

- 一个**唯一的哈希值**（像 `3190df1`，实际是 40 位，平时用前 7 位就够）
- 一条**提交信息**（说明这次改了什么）
- 一个**父提交**（指向上一个存档点，于是所有提交连成一条链）

```
（旧）  A ← B ← C ← D  （新，这就是当前状态）
```

每个字母是一个 commit，箭头指向父提交。

> **SureJack 里的真实例子**：我们阶段 1 做了 36 个提交，比如
> `feat: 断句——标点边界 + 字数上限`、`fix: 进度解析加行缓冲`。
> 每个提交只做一件清晰的事——这是好习惯，后面会讲为什么。

### 1.2 分支（Branch）—— 一个可移动的指针

这是最反直觉、但最重要的一点：**分支不是一堆代码的拷贝，它只是一个「指向某个 commit 的指针」。**

```
                    ┌─ main（指针，指向 C）
                    ↓
      A ← B ← C
                    ↑
                    └─ feat/新功能（另一个指针，也指向 C）
```

刚建分支时，两个指针指向同一个 commit——**没有复制任何代码，只是多了一个标签**。这就是为什么建分支在 Git 里几乎是瞬间完成、零成本的。

当你在 `feat/新功能` 上提交时，只有这个指针往前移动，`main` 纹丝不动：

```
      A ← B ← C           ← main 还在这
                ↖
                 D ← E     ← feat/新功能 移到了 E
```

> **SureJack 里的真实例子**：我们有过三个分支——
> `master`（主线）、`spike/phase0-foundation`（做验证实验的）、
> `feat/phase1-pipeline`（写阶段 1 代码的）。它们只是三个指针。

### 1.3 HEAD —— 「你现在站在哪」

`HEAD` 是一个特殊指针，表示**你当前所在的分支**。`git checkout <分支>` 就是把 HEAD 挪到那个分支上。你的工作目录（你 VS Code 里看到的文件）会变成那个分支指向的 commit 的样子。

```bash
git branch            # 看所有分支，带 * 的那个就是 HEAD 所在
git checkout master   # 把 HEAD 切到 master，工作目录随之变化
```

### 1.4 远程（Remote）—— 云端的另一份仓库

到目前为止说的都在**你自己电脑上**。`remote` 是托管在别处（GitHub、GitLab、公司内网服务器）的**同一个仓库的另一份拷贝**。

- `origin` 是远程仓库的**默认名字**（约定俗成，可以改）
- `git push` = 把你本地的提交**上传**到远程
- `git pull` = 把别人的提交**下载**下来并合并进你的本地
- `git clone` = 第一次把整个远程仓库**下载**到本地

```
你的电脑                          GitHub（origin）
  master ──────── push ────────►  master
  master ◄─────── pull ─────────  master
```

> **SureJack 里的真实例子**：我们刚才做的
> `git remote add origin https://github.com/Zac-Chen-2024/SureJack.git`
> 就是告诉本地仓库「云端那份叫 origin，在这个地址」，
> 然后 `git push -u origin master` 把 36 个提交传上去。

## 2. 为什么要用分支？（feature branch 工作流）

假设你和同事都直接在 `main` 上改代码，会发生什么：

- 你的功能写到一半（代码是坏的），同事要发布，被你半成品坑了
- 两个人同时改同一个文件，push 时疯狂冲突
- 想回滚你的改动，结果和同事的搅在一起分不开

**解决办法：谁干活谁开一条自己的分支。** 这叫 **feature branch workflow（功能分支工作流）**，是当今几乎所有团队的标准做法。

核心规则：

1. **`main`（或 `master`）永远是「随时可以发布」的干净状态**——没人直接在上面改。
2. 每个功能 / 修复，**从 main 分出一条新分支**来做。
3. 做完、测试通过、经过审查，**才合并回 main**。
4. 合并后，**删掉那条功能分支**（它的使命完成了，代码已经在 main 里）。

```
main:     A ── B ─────────────── M ──►   （M 是合并点，功能进来了）
                ╲               ╱
feat/x:          C ── D ── E ──╯          （在这条线上安全地开发）
```

> **SureJack 就是这么做的**：`master` 保持干净，我们在
> `feat/phase1-pipeline` 上写完全部阶段 1（36 个提交），
> 111 个测试全绿、代码评审过了，**才**合并回 `master`。

## 3. 分支命名约定

企业里分支名不是乱起的，有约定，一眼能看出这条分支在干嘛。最常见的前缀：

| 前缀 | 含义 | 例子 |
|---|---|---|
| `feat/` | 新功能（feature） | `feat/user-login` |
| `fix/` | 修 bug | `fix/subtitle-overflow` |
| `hotfix/` | 紧急线上修复 | `hotfix/payment-crash` |
| `docs/` | 只改文档 | `docs/api-reference` |
| `refactor/` | 重构（不改功能） | `refactor/extract-parser` |
| `chore/` | 杂务（依赖升级、配置） | `chore/bump-node-24` |
| `spike/` | 探索性实验（可能会丢弃） | `spike/phase0-foundation` |

> **SureJack 用了两种**：`spike/phase0-foundation`（阶段 0 是验证实验）
> 和 `feat/phase1-pipeline`（阶段 1 是正经功能）。前缀一眼就说明了性质。

## 4. 提交信息约定（Conventional Commits）

提交信息不是写给自己看的日记，是**写给未来的团队（包括三个月后的你自己）看的说明**。业界最流行的规范叫 **Conventional Commits**：

```
<类型>: <一句话说清这次改了什么>

<可选的详细说明：为什么这么改，而不是改了什么>
```

类型和分支前缀类似：`feat` / `fix` / `docs` / `refactor` / `chore` / `test`。

**好的提交信息 vs 坏的：**

```
❌ git commit -m "改了点东西"
❌ git commit -m "fix bug"
❌ git commit -m "update"

✅ git commit -m "fix: 进度解析加行缓冲，避免 chunk 边界切断数字"
```

好的信息回答了「改了什么」，而正文回答「**为什么**」——这个「为什么」是最值钱的，因为代码本身能告诉你「怎么改的」，但说不清「当初为什么要这么改」。

> **SureJack 里的真实例子**（这个提交的正文解释了非改不可的原因）：
> ```
> fix: 进度解析加行缓冲，避免 chunk 边界切断数字
>
> stdout 的 data 事件不保证按行切分。out_time_ms=92100000 若被拆成
> out_time_ms=921 和 00000，正则会匹配到 921 并算出接近 0 的进度，
> 造成进度条倒退。不影响成片，但长视频下会真的触发。
> ```
> 半年后有人看到这段防御代码想删掉时，这条信息会拦住他。

## 5. 一个功能的完整生命周期（把上面串起来）

这是你在企业里每天会重复的循环。假设要做一个「用户登录」功能：

```bash
# ① 从最新的 main 出发
git checkout main
git pull                          # 先同步别人的最新改动，避免基于过时代码开发

# ② 开一条功能分支
git checkout -b feat/user-login   # -b = 新建并切换过去

# ③ 干活，小步提交（重复多次）
#    ...改代码...
git add src/auth/login.ts         # 把改动放进「暂存区」
git commit -m "feat: 登录表单校验"
#    ...再改...
git commit -am "feat: 接入密码哈希"   # -a 自动暂存已追踪文件的改动

# ④ 推到远程（第一次要 -u 建立追踪关系）
git push -u origin feat/user-login

# ⑤ 在 GitHub 上开 Pull Request（见第 6 节）
#    同事审查、你根据意见继续 commit + push

# ⑥ 审查通过，合并到 main（通常在 GitHub 网页上点按钮）

# ⑦ 善后：切回 main，同步，删掉功能分支
git checkout main
git pull
git branch -d feat/user-login     # 删本地分支
git push origin --delete feat/user-login   # 删远程分支（可选）
```

「小步提交」（③）是个重要习惯：**每个提交只做一件能独立描述的事**。这样出了问题能精确定位到哪一步，回滚也只回滚那一步。我们阶段 1 的 36 个提交每个都很小，就是这个原因。

## 6. Pull Request（PR）—— 企业协作的核心

**Pull Request（GitHub 叫 PR，GitLab 叫 Merge Request / MR）是团队协作里最重要的一环。** 它的意思是：

> 「我在我的分支上做完了，**请求**把它合并进 main。合之前，请大家看一眼。」

一个 PR 页面上会有：

- **这条分支相对 main 的所有改动**（diff，绿色是新增、红色是删除）
- **讨论区**：同事逐行留评论（「这里少了个空指针判断」「这个函数命名能不能更清楚」）
- **CI 检查结果**：自动跑的测试、类型检查、代码规范，全绿才能合（见第 9 节）
- **审批状态**：通常要求至少一个同事点「Approve」才能合并

**为什么企业离不开 PR：**

1. **代码审查（Code Review）**：合进主干前有第二双眼睛，能挡掉很多 bug 和坏设计。
2. **知识共享**：审查别人的 PR 是了解项目其他部分最快的方式。
3. **责任留痕**：谁写的、谁审的、为什么这么改，全部有记录，将来出问题能追溯。
4. **保护主干**：配合「保护分支」规则（第 9 节），没人能绕过审查直接往 main 塞代码。

> **SureJack 为什么没走 PR**：因为这是你一个人的项目、目标是把代码存到 GitHub，
> 不需要别人审查，所以我们直接合并到 master 再 push。
> **但在企业里，直接往 main push 通常是被禁止的**——必须走 PR。
> 我们在这个项目里用「派子代理评审每个任务」模拟了 code review 的作用。

## 7. 合并的三种方式（这块最容易懵）

把功能分支合回 main，有三种底层机制。理解它们能帮你看懂 GitHub 上那三个合并按钮的区别。

### 7.1 快进合并（Fast-forward）

**前提**：从你分叉之后，main **没有任何新提交**。那么合并就是简单地把 main 指针「快进」到你的分支顶端——不产生新的提交，历史是一条直线。

```
合并前：
main:   A ── B                    ← main 还在 B
              ╲
feat:          C ── D             ← 你的分支到了 D

合并后（fast-forward）：
main:   A ── B ── C ── D          ← main 直接快进到 D，一条直线
```

> **SureJack 就是这种**：我们合并时 master 是 feat 的祖先（master 之后没动过），
> 所以是干净的快进，输出里那句 `Fast-forward` 就是它。

### 7.2 三方合并（Merge commit）

**前提**：你开发期间，main **也有了新提交**（别人合了东西进去）。这时无法快进，Git 会创建一个**特殊的「合并提交」**，它有**两个父提交**，把两条线重新接起来。

```
main:   A ── B ──── E ── M        ← M 是合并提交，有两个父（E 和 D）
              ╲         ╱
feat:          C ── D ─╯
```

好处：完整保留了「这是一条独立开发的支线」这个历史事实。
坏处：历史图会有很多分叉，复杂项目里看起来像地铁线路图。

### 7.3 变基（Rebase）

**Rebase 是「假装你从最新的 main 开始开发」。** 它把你的提交「摘下来」，接到 main 的最新位置上，historia 变成直线。

```
rebase 前：
main:   A ── B ── E               ← main 前进到了 E
              ╲
feat:          C ── D             ← 你基于旧的 B 开发的

rebase 后：
main:   A ── B ── E               ← main 不变
                    ╲
feat:                C'── D'      ← 你的提交被「移植」到 E 之后（注意 C'/D' 是新哈希）
```

然后再快进合并，得到一条完美直线。很多团队偏爱 rebase 因为历史干净好读。

> **⚠️ Rebase 的铁律：绝不要 rebase 已经 push 出去、别人可能基于它开发的分支。**
> 因为 rebase 会**改写提交哈希**（C 变成 C'），别人的历史会和你对不上，造成灾难。
> 只在自己的、还没分享出去的分支上 rebase。

**新手建议**：先用 merge，它安全、不改写历史。等你完全理解了再碰 rebase。

## 8. 处理冲突（Merge Conflict）

当两条分支**改了同一个文件的同一部分**，Git 没法自动决定听谁的，就产生**冲突**，需要你手动解决。冲突的文件里会出现这样的标记：

```
<<<<<<< HEAD
这是当前分支（比如 main）的版本
=======
这是要合并进来的分支的版本
>>>>>>> feat/user-login
```

**解决步骤：**

1. 打开冲突文件，找到 `<<<<<<<` 标记
2. 决定最终要保留什么——可能是留其中一个，可能是两个都留，可能是重新写
3. **删掉所有 `<<<<<<<`、`=======`、`>>>>>>>` 这些标记行**
4. `git add <文件>` 标记为已解决
5. `git commit`（合并的话）或 `git rebase --continue`（rebase 的话）

冲突不可怕，它只是 Git 在说「这里我不敢替你做主，你来定」。冷静一个个改完即可。

> 想避免冲突：**勤 `git pull`**（常同步别人的改动）、**分支别开太久**（越久越容易和 main 分叉严重）、**功能拆小**（改动范围小，撞车概率低）。

## 9. 企业环境特有的规矩

个人项目和企业项目最大的区别，在于企业加了很多「护栏」：

### 9.1 保护分支（Protected Branch）

`main` 会被设成「保护分支」，强制规则通常包括：

- **禁止直接 push**——必须走 PR
- **PR 必须至少 N 人 Approve** 才能合并
- **CI 必须全绿** 才能合并
- **禁止 force push**（防止有人改写主干历史）

### 9.2 持续集成（CI / Continuous Integration）

你每次 push，服务器会**自动**跑一整套检查：单元测试、类型检查、代码规范（lint）、有时还有安全扫描。全绿才允许合并。

> **SureJack 的对应物**：我们每个任务都跑 `npx vitest run`（111 个测试）
> 和 `npx tsc --noEmit`（类型检查）。企业里这些会被自动化成 CI，
> 你不用手动跑，push 上去它自己跑给你看。

### 9.3 分支模型（Git Flow / GitHub Flow / Trunk-Based）

团队会约定一套分支模型。三种主流：

- **GitHub Flow（最简单，最常见）**：只有一个长期分支 `main`，功能分支从它出、合回它。适合持续部署的产品。
- **Git Flow（较重）**：有 `main`（生产）、`develop`（开发）、`feature/*`、`release/*`、`hotfix/*` 多种分支。适合有明确版本发布周期的软件。
- **Trunk-Based（大厂常用）**：所有人都往一条主干提交，功能分支活得极短（几小时到一天），靠「功能开关」隐藏未完成的功能。

新人进公司第一件事就是问：**「咱们用哪套分支模型？」** 照着做就行。

## 10. 常用命令速查表

```bash
# —— 看状态 ——
git status                  # 当前有哪些改动、在哪个分支
git log --oneline           # 简洁的提交历史
git log --oneline --graph --all   # 带分支图的历史（很有用）
git diff                    # 看还没暂存的改动
git diff --staged           # 看已暂存的改动
git branch                  # 列出本地分支
git branch -a               # 列出所有分支（含远程）

# —— 分支操作 ——
git checkout -b feat/x      # 新建并切换到分支
git checkout main           # 切换分支
git branch -d feat/x        # 删除已合并的分支
git branch -D feat/x        # 强制删除（未合并也删，慎用）

# —— 提交 ——
git add <文件>              # 暂存指定文件
git add .                   # 暂存当前目录所有改动
git commit -m "信息"        # 提交
git commit -am "信息"       # 暂存已追踪文件的改动 + 提交（一步）
git commit --amend          # 修改最后一次提交（还没 push 时用）

# —— 和远程同步 ——
git push                    # 推送到远程
git push -u origin feat/x   # 第一次推送并建立追踪
git pull                    # 拉取并合并远程改动
git fetch                   # 只下载远程改动，不自动合并（更安全）

# —— 合并 ——
git merge feat/x            # 把 feat/x 合并进当前分支
git rebase main             # 把当前分支变基到 main（谨慎）

# —— 救命 ——
git stash                   # 临时把改动收起来（去干别的），git stash pop 取回
git checkout -- <文件>      # 丢弃某文件还没暂存的改动
git reset --soft HEAD~1     # 撤销最后一次提交，但保留改动
git revert <哈希>           # 用一个新提交来「反做」某个历史提交（安全，不改历史）
git reflog                  # 「操作日志」——你几乎做过的所有操作都有记录，是终极后悔药
```

## 11. 常见错误与如何补救

| 情况 | 补救 |
|---|---|
| 提交信息写错了（还没 push） | `git commit --amend -m "新信息"` |
| 提交到了错误的分支 | `git reset --soft HEAD~1` 撤销提交（保留改动），切到对的分支再提交 |
| 想撤销一个已经 push 的提交 | `git revert <哈希>`（用新提交反做，不改写历史，团队安全） |
| 改乱了想回到上次提交的状态 | `git checkout -- .`（丢弃所有未暂存改动，**改动会丢，谨慎**） |
| 「我把什么东西搞没了！」 | `git reflog` 找到那个操作前的哈希，`git checkout <哈希>` 回去 |
| 不小心把大文件/密钥提交了 | **push 之前**：`git rm --cached <文件>` + 改 `.gitignore` + `git commit --amend`；**push 之后**：麻烦得多，需要改写历史 + 立刻吊销泄露的密钥 |

> **最后一条我们真的遇到了**：阶段 1 收尾时，一个自动化步骤 `git add -A`
> 不小心把你的真实素材（37MB 的 mp3）提交了进来。因为**还没 push**，
> 我们用 `git rm --cached` + 更新 `.gitignore` + `git commit --amend`
> 干净地抹掉了，push 上去的仓库里一点痕迹都没有。
> **这就是为什么「push 前检查一遍要提交什么」是个救命习惯。**

---

## 12. 最重要的三条心法

如果这份文档你只记住三句话，记这三句：

1. **`main` 永远保持干净可发布。** 所有开发在功能分支上进行，测试通过、审查通过才合回来。

2. **提交要小而清晰，提交信息要说清「为什么」。** 代码会告诉别人你做了什么，只有提交信息能告诉别人你当初为什么这么想。

3. **push 之前，`git status` 看一眼要传什么；碰不确定的操作之前，想想 `git reflog` 能不能救回来。** 前者防止泄密和事故，后者让你知道 Git 里几乎没有真正无法挽回的操作——这会让你敢于尝试。
